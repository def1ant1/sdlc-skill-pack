# Plan Workspace Developer Guide

## New orchestration modules
- `scripts/orchestration/conversation_to_plan.py`
  - `conversation_to_plan()` pipeline outputting human-readable structured plans first.
  - Living operations: `collapse_expand_section`, `reorder_priorities`, `edit_assumptions`, `approve_step`, `version_diff_history`.
- `scripts/orchestration/plan_to_workflow.py`
  - Converts only approved plan steps into workflow format.
- `scripts/orchestration/plan_to_tasks.py`
  - Converts only approved plan steps into task list format.

## Design notes
- Plan model is dataclass-based (`LivingPlan`, `PlanPhase`, `PlanStep`) to keep mutation logic explicit.
- All living operations increment `version` and append immutable history entries.
- Downstream conversion intentionally filters to `status == approved` to enforce HITL boundaries.


## Planner input contract (MB-P3-008 follow-up)
- `conversation_to_plan.py` now accepts optional `--orchestrator-state <json>` and hydrates prior answers from orchestrator + curated memory (`resolved_questions`, `curated_memory.qa_pairs`).
- `LivingPlan` now carries `planner_inputs` with: `conversation_summary`, `constraints`, `assumptions`, `desired_outputs`, `entities`, `preferences`, and `prior_decisions`.
- `LivingPlan` now carries `confidence_annotations` where each annotation labels whether the detail is `explicit_user` vs `inferred_assumption`.
- `unanswered_questions` is now generated by excluding already-resolved questions, reducing repetitive intake prompts while preserving progressive clarification.
- Existing `phases/tasks/assumptions/objectives` fields are unchanged, preserving compatibility for `plan_to_workflow.py` and graph execution handoff paths.
