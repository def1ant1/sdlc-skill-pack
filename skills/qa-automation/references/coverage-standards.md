# Coverage Standards

## Coverage Philosophy

Coverage is a signal, not a goal. 80% coverage with meaningful assertions on
business-critical paths is more valuable than 100% coverage with trivial assertions.
Coverage gates enforce a floor; quality of test assertions is enforced in code review.

---

## Coverage Metrics

### Line Coverage

Measures which source lines are executed during the test run.
Minimum acceptable floor across all non-generated code.

### Branch Coverage

Measures which conditional branches (if/else, switch cases, ternaries) are executed.
Required for code with complex conditional logic.

### Function Coverage

Measures which functions are called during tests.
Every public function in the domain/service layer must be tested.

---

## Coverage Gate Configuration

```yaml
# coverage-config.yaml
gates:
  - package: "*/domain/*"
    line: 80
    branch: 70
    enforce: block         # CI fails if below
  - package: "*/service/*"
    line: 75
    branch: 65
    enforce: block
  - package: "*/repository/*"
    line: 70
    branch: 60
    enforce: block
  - package: "*/handler/*"
    line: 60
    branch: 50
    enforce: warn          # CI warns but does not block
  - package: "*/generated/*"
    line: 0
    enforce: skip
  - package: "*/cmd/*"
    line: 0
    enforce: skip          # entry points excluded

# Delta gate: new code must not drop coverage by more than:
delta_threshold: 2         # percentage points
delta_enforce: block
```

## What Counts Toward Coverage

Counted:
- Business logic functions
- Service methods
- Repository interface implementations
- Error handling branches
- Conditional flows in domain logic

Not counted (exclude from coverage):
- Generated code (protobuf, OpenAPI clients, mocks)
- `main()` and CLI entry points
- Vendor/third-party packages
- Test files themselves
- Migration files

---

## Coverage Anti-Patterns

| Anti-Pattern | Problem | Fix |
|---|---|---|
| Testing getters/setters for coverage | No behavior is exercised | Remove; cover meaningful logic instead |
| Empty test with no assertions | Coverage metric passes; bugs not caught | Add meaningful assertions |
| Mocking everything in integration test | Integration layer not actually tested | Use real dependencies in integration layer |
| One happy-path test per function | Branches and errors uncovered | Add edge case and error tests |
| Coverage only for new code | Regressions in existing code undetected | Maintain coverage on full changed diff |

## Coverage in CI

```yaml
# .github/workflows/test.yml (excerpt)
- name: Run tests with coverage
  run: |
    go test ./... -coverprofile=coverage.out -covermode=atomic
    go tool cover -func=coverage.out | tail -1

- name: Enforce coverage gates
  run: |
    python scripts/validation/check_coverage.py coverage.out

- name: Upload coverage report
  uses: codecov/codecov-action@v4
```

Coverage reports are:
- Published as PR comment showing delta from base branch
- Archived as CI artifacts for 90 days
- Aggregated to the observability dashboard (weekly trend)

---

## Coverage Debt

If existing coverage is below gate:

1. No new code may be merged that further reduces coverage
2. Coverage debt items are P2 backlog items with a sprint target
3. Teams with coverage < 60% in domain packages require Engineering Manager acknowledgment weekly until resolved
4. Coverage trend (improving/declining) is reported in the weekly engineering review

---

## Excluded Code Patterns

In Go (for example):
```go
// coverage:ignore — generated by protoc
// coverage:ignore — CLI entry point only
```

In JavaScript/TypeScript:
```javascript
/* istanbul ignore next */
/* c8 ignore next */
```

Exclusion comments must include a reason. Reviewer must approve exclusion in code review.
